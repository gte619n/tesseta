# ADR-0009: Exercise demo videos use Veo 3.1 (third exception to the flash-only rule)

- Status: Accepted
- Date: 2026-06-01

## Context

The project convention (root `CLAUDE.md`, "AI Models") runs all general AI work
on **`gemini-3.5-flash`**, reserves **`gemini-3.1-flash-image-preview`** for
image generation, and disallows any other model without an ADR.
[ADR-0005](ADR-0005-goals-chat-gemini-pro.md) carved out Goals chat
(`gemini-3.1-pro-preview`) and [ADR-0007](ADR-0007-workout-program-design-gemini-pro.md)
workout-program design. This record is the third exception.

[IMPL-15](../specs/IMPL-15-workout-programs.md) / [ADR-0008](ADR-0008-workout-history-import.md)
seed an exercise catalog whose demo media so far is **still frames**
(START/MID/END) from the approved image model. `Exercise.videoUrl` was always
reserved "for future Veo." We now want short **looping demo videos** of the
movement — a strictly better teaching aid than three stills — starting with a
reviewed sample set before any catalog-wide rollout.

There is no still-image model that produces video. Animating a movement demo
requires a video-generation model, and Google's is **Veo**. The bundled GenAI
SDK (`com.google.genai` 1.53.0) already exposes the Veo API
(`Models.generateVideos`, `Operations.getVideosOperation`, `Files.download`),
so no new dependency or provider is introduced — only a new **model**.

## Decision

Use **`veo-3.1-generate-preview`** (Veo 3.1, standard quality) for exercise
demo-video generation, **image-to-video** seeded by the approved still:
- The START demo frame (already generated by `gemini-3.1-flash-image-preview`)
  is the seed image, so the video matches the still library's subject, wardrobe,
  and seamless-studio treatment (see `docs/photography-prompts.md`).
- A short motion prompt describes the rep (start → mid → end → return).
- Output is portrait (`9:16`) to match the catalog's vertical framing.

Model id is overridable via env (`VEO_MODEL`) so the exact preview/GA id can be
swapped without a code change.

## Why this is the right exception

- **No flash alternative.** Unlike text tasks, there is no `gemini-3.5-flash`
  path to video; the capability only exists in Veo.
- **Same provider, same SDK, same API key.** Veo runs through the existing
  `com.google.genai` client and `GEMINI_API_KEY`; this widens the *model* set,
  not the dependency or vendor surface.
- **Human-gated, not in the hot path.** Like demo images, generated video is a
  reviewed asset (admin approval), produced by an offline job — never on a user
  request path.

## Cost & scope guardrails

Veo is billed per second of video and is materially more expensive and slower
than image generation. Therefore:
- **Samples first.** Generation starts as a reviewed sample set
  (`docs/test_reports/workout_logs/media_preview/*.mp4`), gated on
  `GEMINI_API_KEY` and run manually — never in CI.
- **Catalog-wide rollout is a separate, explicit step** (a future bounded
  backfill job mirroring `BackfillExerciseMediaJob`), not enabled here.
- Videos remain optional enrichment; the still frames stay the baseline demo.

## Consequences

- `Exercise.videoUrl` now has a real producer path (sample stage).
- A future `BackfillExerciseVideoJob` + storage wiring is needed before
  catalog-wide video; deferred until the sample quality is approved.
- A fourth non-flash model would, per the standing convention, warrant its own
  ADR rather than widening this one.
