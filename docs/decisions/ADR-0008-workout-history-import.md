# ADR-0008: Importing real workout history from future_workouts.json

- Status: Accepted
- Date: 2026-06-01

## Context

`docs/test_reports/workout_logs/future_workouts.json` is a real training export
— a 352-entry name-only exercise catalog plus a flat log of 411 completed
sessions (2023-03 → 2025-11) tagged across 20 phases. We want it in Firestore as
the basis for the workout-programs feature ([IMPL-15](../specs/IMPL-15-workout-programs.md)):
a populated catalog, the user's phase history, and the per-set weights actually
lifted.

The data has three properties that drove the decisions below:

- **Weight-only logging.** Every set records `weight_lbs` only; all `reps` are
  null. Some weights are `NaN` ("no weight recorded").
- **No weekly template.** `workout_name` is mostly a unique motivational title
  per session (Phase 1: ~130 distinct names over 142 sessions), so there is no
  recurring weekly microcycle to recover — only a phase-tagged session log.
- **Ids are canonical, names drift.** The log uses 395 distinct `exercise_name`
  spellings over 352 ids; the id is the key and maps 1:1 to the history.

## Decisions

1. **Owner.** Programs, sessions, and history are per-user; the import is run
   for one real user whose id (`sub`) is supplied at job time
   (`app.workouts.seed.user-id`) and never committed.

2. **Exercise ids preserved.** Each source exercise is written as an `Exercise`
   keyed by its **source UUID** (via `ExerciseRepository.save`, not
   `ExerciseService.create`, which would mint new `ex_…` ids and break the
   log's references). `PUBLISHED`, no media.

3. **AI enrichment.** Source exercises carry only a name; gym-availability and
   image prompts need structured metadata. A new `ExerciseMetadataEnricher`
   port (`GeminiExerciseMetadataEnricher`, **`gemini-3.5-flash`** — the approved
   general-work model, no new exception needed) fills movement pattern, muscles,
   laterality, mechanic, form cues, suitable block types, rep range, and
   equipment-by-name (resolved to catalog ids, unresolved reported).

4. **History via a new `loggedSets` field.** No existing model stored performed
   sets (`Prescription` is a plan; `ScheduledWorkout.session` is a snapshot;
   `Workout` is session-level only). We added `LoggedSet(weightLbs, reps)` and
   `Prescription.loggedSets` (null for plans, populated in completed snapshots).
   Each session becomes a `COMPLETED` `ScheduledWorkout`; `NaN` weights are
   sanitized to null; reps stay null (nothing invented).

5. **Phases, no templates.** One `WorkoutProgram` (`COMPLETED`, id
   `imported-history`) holds 20 date-bounded `ProgramPhase`s (ordered by first
   session date, `days = []`); sessions link by `phaseId`. We did **not**
   synthesize weekly day-templates — the source has none.

6. **No dedup.** All 352 ids are treated as canonical (1:1 with the history;
   near-duplicates are mostly real variants). The catalog's `aliasOfExerciseId`
   / `ExerciseService.mergeInto` remains for ad-hoc merges later.

7. **Images now, video deferred.** Demo stills are generated by the existing
   `GeminiExerciseMediaService` (`gemini-3.1-flash-image-preview`), landing
   `NEEDS_REVIEW` for admin approval. Exercise **video** is not built —
   `Exercise.videoUrl` is reserved for a future Veo integration, which is not in
   the approved model list and would need its own ADR.

## Mechanism

`WorkoutHistoryImporter` (core, pure/testable) performs the mapping. Two Cloud
Run Jobs drive it on the shared backend image, mirroring the IMPL-13 food
seeding (`SeedFoodCatalogJob` / `BackfillFoodImagesJob`):

- `SeedWorkoutHistoryJob` (`@Profile("job-seed-workouts")`) — parse + import +
  enrich. `infra/scripts/deploy-seed-workouts-job.sh`.
- `BackfillExerciseMediaJob` (`@Profile("job-exercise-media-backfill")`) —
  generate demo media for exercises at `mediaStatus=NONE`, bounded per run.
  `infra/scripts/deploy-exercise-media-job.sh`.

Both are idempotent (deterministic ids; catalog skips existing ids).

## Verification

`WorkoutHistorySeedMappingTest` runs the importer over the real file with
in-memory repos and a deterministic fake enricher, asserts the invariants
(catalog count, 20 date-ordered phases, 411 sessions, logged-set fidelity, reps
stay null), and writes `docs/test_reports/workout_logs/seed_preview.json` — the
exact Firestore document shapes — for review before the real job runs.
`WorkoutSeedEnrichmentPreviewTest` (gated on `GEMINI_API_KEY`) previews real
enrichment quality.

## Consequences

- The web service now constructs the Gemini enricher at startup (like
  `EquipmentParserService`); prod has the key, tests gate it off
  (`app.exercises.enrich-enabled=false` + fake bean).
- `loggedSets` is wired through Firestore and the deep response DTO; surfacing
  it in the web/Android clients is a follow-up.
- Enrichment quality is only as good as the model from a bare name; the
  unresolved-equipment report and the `NEEDS_REVIEW` media gate keep a human in
  the loop.
