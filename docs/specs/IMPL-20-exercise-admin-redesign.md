# IMPL-20: Exercise admin redesign — tiles, filters, reviewed flag, own-image grounding, thumbnails

## Goal of this spec

The exercise admin catalog ([IMPL-14](IMPL-14-exercise-library.md)) was built
for a fixed three-image triad and a flat searchable list. [IMPL-19](IMPL-19-dynamic-demo-frames.md)
made the image count **per-exercise dynamic** (1–5 frames), which the current
list UI renders awkwardly. This spec reworks the **admin experience** for the
352-exercise catalog and adds two new capabilities:

1. **A browsable catalog with two view modes** — keep today's list, add a tile
   grid, toggle between them like a file browser — backed by a shared
   filter/sort layer (status, media status, plan status, reviewed, image count,
   movement pattern, workout/block type, name).
2. **A human `reviewed` sign-off** distinct from the existing pipeline gates,
   and **own-image grounding** — the admin can pick which images (the
   exercise's own generated/uploaded candidates *and/or* its external library
   references) are fed back into regeneration as pose references.

Plus infra: a **Cloud Function** that auto-generates GCS thumbnails for exercise
media on upload (see [ADR-0017](../decisions/ADR-0017-gcs-thumbnail-cloud-function.md)).

This is backend + web + infra. No Android, no user-facing client change.

## Decisions locked

| Decision | Choice |
|---|---|
| List vs grid | **Both** — a Finder-style view toggle over one shared filter/sort/data layer; default grid, persisted per-admin in `localStorage` |
| "Reviewed" | **New separate boolean** `reviewed` on `Exercise` — a whole-exercise human sign-off, independent of `mediaStatus`/`planStatus` |
| Reference images | **Unified, persisted picker** — own GCS candidate URLs *and* external `reference.images` URLs, each toggleable; stored as `groundingImageUrls`, overridable per-regen |
| Variable image count | **No data change** — [IMPL-19](IMPL-19-dynamic-demo-frames.md) `demoPlan`/`demoFrames` already model it; this is a rendering concern |
| Catalog scale | **352 today ⇒ client-side filter/sort** over a slimmed summary projection; image-thin tiles use thumbnails |
| Thumbnails | **Cloud Function Gen2**, GCS-finalize triggered, derive-by-convention path, no Firestore write ([ADR-0017](../decisions/ADR-0017-gcs-thumbnail-cloud-function.md)) |
| Review tab | **Folded into a "Needs review" filter preset** (route kept as a redirect); works in either view |

## Concepts

**View mode** — `list` or `grid`, a pure rendering choice over the same
filtered/sorted array. Switching is instant and stateless; filters, search, and
selection carry across. Persisted in `localStorage` (no backend).

**Reviewed** — a boolean the admin sets to mean "I have looked at this whole
exercise and it's good." Deliberately *not* coupled to `mediaStatus` (per-image
pipeline) or `planStatus` (frame-plan pipeline); an exercise can have
`APPROVED` media and still be un-`reviewed`, or vice-versa.

**Grounding image set (`groundingImageUrls`)** — the persisted list of image
URLs that regeneration uses as pose references. Entries are either own GCS
candidate URLs (`storage.googleapis.com/<exercise-media-bucket>/…`) or external
`reference.images` URLs. This generalizes [IMPL-19](IMPL-19-dynamic-demo-frames.md)
grounding, which was external-library-only and transient.

**Thumbnail** — a small webp derivative of each stored exercise image,
generated by a Cloud Function and addressed by path convention
(`exercises/{id}/thumb/{name}.webp`). Not persisted on the document; the web
derives the URL and falls back to the full image on miss.

## Data model

Two additive fields on `exercises/{exerciseId}` (nothing removed; `merge()`-safe):

```
reviewed: boolean                      default false   human whole-exercise sign-off
groundingImageUrls: string[]           default []      images used as regen pose references
```

`groundingImageUrls` entries are full URLs — own GCS candidate URLs and/or
external `reference.images` URLs. No thumbnail field: thumbnails are
path-derived (see ADR-0017).

## Backend — Spring

### Records & mapping

`Exercise` gains `boolean reviewed` and `List<String> groundingImageUrls`.
`FirestoreExerciseRepository.toExercise`/`toBody` and `ExerciseResponse` extend
accordingly; defaults make legacy docs read cleanly.

### Slim catalog projection

`GET /api/admin/exercises/catalog?view=summary` returns only what the
list/grid need, so the 352-item payload stays light and image-thin:

```
ExerciseSummaryResponse {
  exerciseId, name, movementPattern, suitableBlockTypes, primaryMuscles,
  status, mediaStatus, planStatus, reviewed,
  frameCount: int,                     # demoPlan.size (or demoFrames.size legacy)
  frameImageUrls: string[]             # active imageUrl per frame, in order (may contain nulls)
}
```

The existing full `GET /catalog` (no param) and `GET /{id}` stay for detail.
Web derives thumbnail URLs from `frameImageUrls` client-side.

### REST endpoints (admin, `/api/admin/exercises`)

| Endpoint | Method | Purpose |
|---|---|---|
| `/catalog?view=summary` | GET | slim list projection for the catalog grid/list |
| `/{id}/reviewed` | POST | set the `reviewed` flag — `{ reviewed: boolean }` → `ExerciseResponse` |
| `/{id}/grounding` | PUT | persist the grounding set — `{ imageUrls: string[] }` → `ExerciseResponse` |

`RegenerateMediaRequest` gains an optional `referenceImageUrls: string[]`. The
effective grounding set for a regeneration is
`referenceImageUrls ?? groundingImageUrls` (request overrides the persisted
default; omitted ⇒ use persisted; empty list ⇒ explicitly none).

### Grounding rework (`GroundingImageResolver`, `GeminiExerciseMediaService`)

- `GroundingImageResolver` gains an **own-object branch**: a URL pointing at the
  exercise-media bucket is fetched directly from GCS (bytes); external URLs keep
  IMPL-19's per-source resolution (fedb / Wikimedia / page scrape).
- The generator resolves the **effective grounding set** (above) instead of only
  `reference.images`. Existing frame→image mapping (first→first, last→last,
  interior→proportional) is reused; per-frame assignment stays out of scope.
- Unchanged: resolved bytes are **never persisted**; only the *selection* of
  URLs is stored. Output frames remain house-style and ours.

## Web — Next.js

`AdminExerciseCatalog.tsx` splits into a shared container plus three pieces:

- **`CatalogToolbar`** — search, sort (name / recently updated / image count),
  the filter bar, and the **list ⇄ grid view toggle** (segmented control,
  `localStorage`-persisted).
- **`ExerciseListView`** — today's expandable rows, kept; gains a `reviewed`
  checkbox column and is fed by the shared filtered array.
- **`ExerciseGridView`** — new tiles: name, **categories** (movement pattern +
  block/"workout" type pills), a **thumbnail strip** of the active frames
  (variable count, thumbnails with `onError` fallback to full image), and
  status + `reviewed` badges. A `reviewed` checkbox on each tile (optimistic →
  `/{id}/reviewed`).

Both views consume the slim `view=summary` payload and the same in-memory
filtered/sorted list; clicking a row or tile opens the same detail drawer
(`ExerciseDemoFrames` + regen flow), which lazy-loads full detail via `GET /{id}`.

**Filters (client-side):** status, mediaStatus, planStatus, reviewed (yes/no),
image count (0 / 1 / 2 / 3+), movement pattern, block/workout type, free-text
name. The **Review tab** becomes a "Needs review" preset
(`mediaStatus = NEEDS_REVIEW || planStatus = NEEDS_REVIEW`); `/admin/exercises/review`
redirects to the catalog with that preset applied.

**Unified reference picker** (in `RegenerateMediaModal.tsx` / `ExerciseDemoFrames.tsx`):
shows the exercise's own candidate thumbnails *and* its external
`reference.images`, each with an include/exclude toggle. Saving writes
`groundingImageUrls` via `PUT /{id}/grounding`; the regen call sends the
selected set as `referenceImageUrls`.

`web/lib/types/exercise.ts`: add `reviewed` and `groundingImageUrls` to the
exercise types; add `ExerciseSummaryResponse`. `web/lib/exercise-admin-api.ts`:
add the summary fetch, reviewed toggle, and grounding save.

## Infra — thumbnail Cloud Function

See [ADR-0017](../decisions/ADR-0017-gcs-thumbnail-cloud-function.md). In brief:

- **Trigger:** GCS `object.finalized` on `health-fitness-160-exercise-media`.
- **Runtime:** Node 20 + `sharp`; output webp, max edge 320px, headers
  `public, max-age=31536000, immutable`.
- **Path:** `exercises/{id}/{name}.{ext}` → `exercises/{id}/thumb/{name}.webp`.
- **Loop guard:** skip any object whose path contains `/thumb/`. Covers both
  AI-generated and admin-uploaded images (both land in this bucket).
- **No Firestore write** — web derives the thumb URL by convention.
- **Code:** `functions/exercise-thumbnails/`; **deploy:** `infra/scripts/deploy-thumbnail-fn.sh`
  (mirrors the per-job deploy scripts); the function SA gets `storage.objectAdmin`
  on the bucket. First Cloud Function in the repo — flagged in the PR.

## Build sequence

1. **Backend schema.** `reviewed` + `groundingImageUrls`, repository mapping,
   `ExerciseResponse`, the slim `ExerciseSummaryResponse` + `view=summary`.
2. **Backend endpoints.** `/{id}/reviewed`, `/{id}/grounding`, and the
   `referenceImageUrls` extension to regenerate-media.
3. **Grounding rework.** Own-object branch in `GroundingImageResolver`;
   effective-set resolution in the generator.
4. **Thumbnail function.** `functions/exercise-thumbnails/` + deploy script +
   SA grant; backfill existing images.
5. **Web.** Toolbar + view toggle + filters; grid view + tiles + thumbnails;
   reviewed checkbox; unified reference picker; Review→preset redirect.

## Out of scope

- **Per-frame** grounding assignment (the picker is a per-exercise pool).
- **Client (user/Android) consumption of thumbnails** — admin tiles only for now;
  the convention leaves room to adopt them in client lists later.
- **Server-side filtering / pagination** — revisit only if the catalog outgrows
  comfortable client-side handling (well above 352).
- Any change to the [IMPL-19](IMPL-19-dynamic-demo-frames.md) plan/media
  pipelines beyond the grounding source.

## Acceptance criteria

- The catalog renders as either a list or a tile grid; toggling is instant,
  preserves filters/search/selection, and the choice persists across reloads.
- Tiles show name, categories, and the exercise's actual frame images (variable
  count) via thumbnails, falling back to full images on miss.
- Filtering by status, media/plan status, reviewed, image count, movement
  pattern, and workout type works in both views.
- A `reviewed` checkbox toggles and persists, independent of `mediaStatus`/`planStatus`.
- The reference picker lets the admin include/exclude own candidates and
  external references; regeneration uses exactly the selected set; resolved
  bytes are never persisted.
- New exercise images in GCS get a thumbnail automatically (generated or
  uploaded); the function never reprocesses its own output.
- Legacy documents (no `demoPlan`, `reviewed` absent) read and render unchanged.
- No new AI model or provider is introduced.
